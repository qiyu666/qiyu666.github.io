ReentrantLock 源码分析

本文最后更新于：2022年11月22日 下午

简介

ReentrantLock 是一种基于 AQS 实现的可重入互斥锁，和 synchronized 类似，通过独占共享资源和阻塞等待队列实现

可重入，即同一个线程在进入某个方法前获取到了锁，在进入方法后如果还需要这把锁，会自动获取，不会因为前一个锁还没释放而阻塞线程，不过前提是获取的是同一把锁，一定程度上避免了死锁

互斥，即同时只能有一个线程占有此锁

使用方法

在源码的开头的介绍中，官方推荐通过 try/catch 将其包起来，并总是在 finally 中调用 unlock() 方法释放锁，避免因为一些异常状况导致某个线程一直持有该锁无法释放

构造方法

ReentrantLock 有两个构造方法，无参构造默认会创建一个 非公平的同步锁 ，而有参构造会根据传入的 fair 来决定使用哪种锁

FairSync 和 NonfairSync ，都是 ReentrantLock 的内部类 前者是公平锁的实现，后者是非公平锁的实现

公平锁，所有线程都老老实实的排队

非公平锁，在排队前，先去抢一次锁，抢到了就直接执行，否则老老实实去排队

两者都继承自 ReentrantLock 的另一个内部类 Sync，Sync 又继承自 AQS

FairSync 和 NonfairSync 重写了 AQS 的 tryAcquire() 方法，Sync 重写了 tryRelease() 方法，通过重写这两个方法来实现一个简单的互斥锁

非公平锁

抢锁

通过 new ReentrantLock() 默认创建一个非公平锁，然后调用 lock() 方法来获取锁，lock() 方法是 Sync 类里的一个抽象方法，由子类实现

从非公平锁实现的 lock() 方法可以看出和公平锁的本质区别，先不管三七二十一，通过 CAS 操作将 state 设置成 1 来抢一次锁，由于这个操作要保证内存可见性和原子性，所有这个 state 是 volatile 修饰的，且在抢锁的时候修改，都采用的 CAS 操作

state，是 AQS 中用于记录锁同步状态的属性，0表示没有上锁，否则等于几就表示重入了几次。

比如，如果同一个线程重入两次，则 state = 2

如果设置成功，则说明抢到了锁，再通过 setExclusiveOwnerThread(Thread.currentThread()) 将当前锁的占有者设置成当前线程，

setExclusiveOwnerThread()，用于设置当前锁的占有线程，

这个方法定义在 AQS 的父类 AbstractOwnableSynchronizer 中，这个类只做一件事，设置锁的独占线程

排队 && 阻塞

如果没有抢到，则会调用父类 AQS 的 acquire() 方法，以独占模式抢占锁，来看一下这个方法都干了些什么

首先会通过 tryAcquire() 方法再次尝试获取

注意这个是 NonfairSync 中的方法，这个方法来自 AQS 的一个模板方法

然后有调用了一个 nonfairTryAcquire() 方法，大体步骤如下

  • 判断当前是否有线程占着锁（c==0），如果没有，会再一次尝试枪锁，抢到了直接将其设置成锁的占有线程
  • 如果有线程占着锁，判断是否和当前线程相等，相等的话将 state 加1
  • 以上都不满足，则说明锁获取失败，当前线程需要进行排队

如果当前锁已经被占有，且不是当前线程，则该线程需要进行排队，即会执行 acquireQueued(addWaiter(Node.EXCLUSIVE), arg) 这行代码

先看 addWaiter() 方法，见名识意，添加等待者，即将当前线程包装成 Node 然后添加到队列尾部，具体操作如下

如果 tail 为空或者设置尾结点失败，会进入 enq() 方法，为什么会存在空的情况呢？看这段源码上的注释

CLH queues need a dummy header node to get started. But we don’t create them on construction, because it would be wasted effort if there is never contention. Instead, the node is constructed and head and tail pointers are set upon first contention.

大概意思就是 CLH 队列需要一个虚拟头结点来启动队列，但只有在需要的时候才会进行初始化，也就是第一次发生争抢的时候会进行初始化，并设置头结点和尾结点，否则不会进行初始化，也就是说这时头尾结点都等于 null

经过上述两个方法完成了等待节点的插入，接下来进入 acquireQueued() 方法进行自旋判断自己是不是头结点的后继，来进行一些后续操作

  • 判断自己是不是 head 的后继，是的话会尝试抢一次锁，抢到了直接运行，无需挂起

  • 如果不是 head 的后继或者没抢到锁，会先检查其前驱是不是 SIGNAL 状态

    如果节点的等待状态为SIGNAL，那么当它释放同步资源时，将会唤醒离它最近的同时等待状态不为CANCELLED的后继节点（同时也能说明节点存在后继节点）

    • 如果是，挂起
    • 如果不是，会判断前驱是不是处于 CANCELLED 状态
      • 是，往前寻找直到找到一个没有处于这个状态的节点
      • 不是，将前驱的状态设置成 SIGNAL 状态，并挂起线程

  • 如果发送异常，且没抢到锁，会调用 cancelAcquire() 将当前节点的状态设置为 CANCELLED ，并且剔除

最后，如果 acquireQueued() 方法里的 for 第一次执行就拿到了锁，那么无需阻塞线程，否则会阻塞当前线程

公平锁

抢锁

公平锁在运行 lock() 方法的时候不会去抢锁，这是和非公平锁最本质的区别

线程都是老老实实的调用 acquire() 方法排队等候，这个方法和非公平锁一样，只是在尝试获取锁的时候有点不同

  • 先判断当前是否有线程占着锁(c==0)，如果没有会先看当前队列的头结点有没有后继

    • 如果后继不存在( h != t 不成立)，会去抢锁
    • 如果后继存在( h != t 成立)，还要看当前线程和后继所代表的线程是否一致( s.thread != Thread.currentThread() 是否成立)，再决定要不要去抢锁

  • 如果有线程占着锁，判断是否和当前线程相等，相等的话将 state 加1

  • 以上都不满足，则说明锁获取失败，当前线程需要进行排队

排队，阻塞和非公平锁一致，不过多赘述

释放锁 && 唤醒后继

因为公平锁和非公平锁释放锁实现都是一样的，所以单独提出来分析

在使用方法这一小节，提到了调用 unlock() 方法来释放锁，在执行 unlock() 方法时，会进入 release() 方法

  • 开头通过 tryRelease() 方法来判断是否可以释放锁，这也是来自 AQS 的一个模板方法

    • 如果上述方法 return true 进入 if ，然后再判断头结点是否存在且状态是否处于非活跃状态，来决定是否要唤醒后继

      • 如果 h == null ，当前队列还未进行初始化，也就是当前只有一个线程，所以自然不需要唤醒

      • 如果 h.waitStatus == 0 ，说明 h 的后继节点还处于活跃状态，可以通过 tryAcquire() 方法进行抢锁，只有 h 的状态处于 SIGNAL 是，后继节点才会被挂起，从而需要被唤醒，否则无需唤醒

      • 以上都不满足，则需要进行唤醒操作
