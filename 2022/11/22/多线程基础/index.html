进程和线程

线程就是独立的执行路径，进程包含多个线程

  • 进程是执行程序的一次执行过程，是一个动态的概念，也系统分配资源的基本单位。线程则是任务调度和执行的基本单位
  • 在程序运行是，即使没有自己创建线程，后台也会有多个线程，如main主线程，gc线程
  • 在一个进程中，如果创建了多个线程，那么这些线程的运行由调度器安排调度，调度器是与操作系统紧密相关的，先后顺序不可人为干预
  • 对同一份资源进行操作是，会存在资源抢夺的问题，因此需要加入并发控制
  • 线程会带来额外的开销。如cpu调度时间，并发控制，且线程都在自己的工作内存中交互，内存控制不当会导致数据不一致

上下文切换

当一个线程进行到某个时候被挂起，由操作系统切换到另一个线程，那么这个被挂起的线程运行时产生的数据就称为上下文，需要进行保存，从而保证下次切换到这个线程时，能够衔接上继续运行

被挂起的线程在运行时的状态称为用户态，线程的调度需要由操作系统进行，这个调度过程称为内核态

happens-before

happens-before那些事儿

happens-before是一些规则，规定了某个线程修改的变量何时对其他线程可见。从JDK 5 开始，JMM就使用happens-before的概念来阐述多线程之间的内存可见性。在JMM中，如果一个操作执行的结果需要对另一个操作可见，那么这两个操作之间必须存在happens-before关系，不过存在这种关系的两个操作并非一定按照其规则规定的顺序进行，如果重排序之后的执行结果和按 happens-before 规定的顺序的执行结果一致，那么这个重排序也合理

happens-before 的规则：

  • 程序顺序规则：一个线程中的每个操作，happens- before 于该线程中的任意后续操作。
  • 监视器锁规则：对一个监视器锁的解锁，happens- before 于随后对这个监视器锁的加锁。
  • volatile 变量规则：对一个 volatile 域的写，happens- before 于任意后续对这个 volatile 域的读。
  • 传递性：如果 A happens- before B，且 B happens- before C，那么 A happens- before C。
  • 线程启动规则：假定线程A在执行过程中，通过执行ThreadB.start()来启动线程B，那么线程A对共享变量的修改在接下来线程B开始执行后确保对线程B可见。
  • 线程终结规则：假定线程A在执行的过程中，通过制定ThreadB.join()等待线程B终止，那么线程B在终止之前对共享变量的修改在线程A等待返回后可见。

内存屏障和volatile

关键字: volatile详解

由于编译器和处理器会对指令进行重排，重排之后可能会导致一些逻辑上的问题，那么为了规避这类问题，就有了内存屏障

内存屏障是一种 CPU 指令，通过在读操作或者写操作的前或者后插入这种指令，可以避免与其前后指令进行重排序

在 Java 中可以使用 volatile 关键字来保证一个变量在一次读写操作时，其指令不被重排，因为会在 volatile读写操作前后加上 StoreStore 屏障 、StoreLoad 屏障 、LoadLoad 屏障、LoadStore 屏障 这几张屏障

内存可见性

当被 volatile 关键字修饰的变量进行写操作后，会强制将其刷到主内存中，并将其他缓存的设置成无效，所以 volatile 可以解决内存可见性的问题

内存可见性：在多线程环境下，前一个指令的执行结果，能不能被后一条指令读取到，能则称其是内存可见，反之为内存不可见

CAS和AQS

并发三大特征

  • 原子性，一个或多个操作，其操作执行顺序不可打乱，且不可拆分或者只执行其中的某些操作，整个操作完了之后，要么成功，要么失败
  • 可见性，在多线程环境下，前一个指令的执行结果，能不能被后一条指令读取到，能则称其是内存可见，反之为内存不可见
  • 有序性，指令被重排序问题

CAS

CAS（compare and swap），通过当前值和预期值进行比较来决定这次操作是否成功，如果不成功将通过自旋不断的去尝试直到成功，从而保证操作原子性

CAS 的缺点

  • 开销问题

如果在线程不多的情况下，省去了挂起线程所带来的一系列操作，但是如果有很多个线程并发执行，自旋可能会长时间不成功，这样会加大 CPU 的开销

  • ABA 问题

ABA problem

“In multithreaded computing, the ABA problem occurs during synchronization, when a location is read twice, has the same value for both reads, and “value is the same” is used to indicate “nothing has changed”. However, another thread can execute between the two reads and change the value, do other work, then change the value back, thus fooling the first thread into thinking “nothing has changed” even though the second thread did work that violates that assumption.

The ABA problem occurs when multiple threads (or processes) accessing shared data interleave. Below is a sequence of events that illustrates the ABA problem:

  1. Process P1 reads value A from some shared memory location,
  2. P1 is preempted, allowing process P2 to run,
  3. P2 writes value B to the shared memory location
  4. P2 writes value A to the shared memory location
  5. P2 is preempted, allowing process P1 to run,
  6. P1 reads value A from the shared memory location,
  7. P1 determines that the shared memory value has not changed and continues.

Although P1 can continue executing, it is possible that the behavior will not be correct due to the “hidden” modification in shared memory.”

解决办法：增加版本号，通过比对版本号来决定是否操作成功

  • 只能保证一个变量的原子操作

AQS

AQS(AbstractQueuedSynchronizer)，即抽象同步队列，通过维护一个名为 CLH(Craig, Landin, and Hagersten) 的双向链表来实现

CLH，一个 FIFO 的双向虚拟队列(即不存在队列实例，仅存在结点之间的关联关系)，每个节点都等待其前驱节点释放锁之后将自己唤醒

AQS核心思想是，如果被请求的共享资源空闲，则将当前请求资源的线程设置为有效的工作线程，并且将共享资源设置为锁定状态。如果被请求的共享资源被占用，那么就需要一套线程阻塞等待以及被唤醒时锁分配的机制，这个机制AQS是用CLH队列锁实现的，即将暂时获取不到锁的线程封装成一个 CLH 锁队列的一个结点（Node）加入到队列中，从而来实现锁的分配

其内部通过 private volatile int state 来记录锁的状态，由于这个操作要保证内存可见性和原子性，所有是 volatile 修饰的，且在抢锁的时候修改，都采用的 CAS 操作

其原理如下图

锁

synchronized

synchronized 同步锁，可以修饰实例方法，静态方法，代码块

修饰实例方法，给当前实例加锁

修饰静态方法，给当前类对象加锁

修饰代码块，给指定的对象加锁

无锁，偏向锁，轻量级锁，重量级锁

无锁：不上锁

偏向锁：偏向某一个线程，被偏向的那个线程访问时不需要获取锁

轻量级锁：少量线程，先尝试自旋（默认10次）来获取锁，而不是直接挂起

  • 线程的挂起和恢复涉及到用户态内核态的切换，上下文的保存等，但是在线程不多的情况下，为了去等上一个线程执行完而且挂起，不如通过少量循环来不断尝试获取锁，这个循环的过程又称为 自旋，所以这个锁也被称之为 自旋锁。

重量级锁：线程排队等待执行

锁升级的过程

偏向锁、轻量级锁和重量级锁

浅谈偏向锁、轻量级锁、重量级锁

无锁 -> 偏向锁 -> 轻量级锁 -> 重量级锁

刚开始没用线程占有这个锁，那么无需上锁，处于无锁状态。后续如果有一个线程来申请这个锁，那么这个锁会升级成 偏向锁，当这个线程再次来获取时，无需在做任何工作，直接获得锁。否则如果有另外一个线程来争抢这个锁，那么锁会进一步升级成 轻量级锁，在这之后，如果有线程来竞争锁，先判断是否是 重入，如果是会添加一条 Displaced Mark Word 为空的 Lock Record，否则会先尝试 自旋 获取锁，如果自旋也未能获取到会再进一步膨胀成 重量级锁，并且阻塞线程。此后所有的线程来了都需要挂起等待

注意：锁只能一级一级往上升级，而不能往下降级

乐观锁，悲观锁

乐观锁，总是认为当自己在读某个数据时，不会有其他人来修改这个数据，因此不会加锁，即非阻塞同步，更适合读操作多的场景，而悲观锁则相反，总是认为会有人在自己读时，修改某个共享数据，因此会上锁，即阻塞同步，更适合写操作多的场景

自旋锁，适应性自旋锁

自旋锁：当只有少量线程的时候，为了等前一个线程释放锁而去挂起线程和恢复线程，不如让其稍微等等(自旋)，在等的过程中，再尝试去获取锁(默认10次)，如果在这10次内还没有获取到，将挂起

适应性自旋锁：根据上一次自旋是否获取到锁，自旋时间等来决定这次的自旋时间或者不进行自旋操作直接挂起

公平锁，非公平锁

公平锁：多个线程按照排队的顺序一个个获取锁，这样做的好处是能保证不会出现某个线程因抢不到锁而长时间阻塞，但是整体效率相比非公平锁较低，因为所有的线程都需要排队等待CPU进行调度

非公平锁：多个线程排队等待锁，这时新来的会先尝试去获取锁，如果能获取到则直接运行，无需阻塞，这样整体效率相比公平锁较高，因为线程有几率不阻塞直接运行，但会导致某个线程因抢不到锁而长时间阻塞

可重入锁，不可重入锁

可重入锁：同一个线程在进入某个方法前获取到了锁，在进入方法后如果还需要这把锁，会自动获取，不会因为前一个锁还没释放而阻塞线程，一定程度上避免了死锁的情况，不过前提是获取的是同一把锁

不可重入锁：与可重入锁相反，再方法内还需要锁的时候，要等前一个释放锁，才能进行获取，容易造成死锁

排他锁，共享锁

排他锁(简称X锁，又称写锁，独享锁)：一个线程对某个数据上了排他锁之后，只允许这个线程对这个数据进行读写，其他线程不允许再上任何类型的锁以及操作这个数据，只能等这个线程释放

共享锁(简称S锁，又称读锁)：可以被多个线程共有，但如果一个线程对某个数据上了共享锁，那么其他线程也只能上共享锁，获取锁的线程只能进行读操作，这也就意味着，加了共享锁的数据是可以多个线程同时进行的

死锁

当多个线程同时被阻塞，它们当中的一个线程或者全部都在等待其中某个正在执行的线程释放资源，但是这个线程被无限期的阻塞，这种场景被称为死锁

产生死锁的条件

四个必要条件

  • 互斥，一个资源只能被一个线程占用
  • 不可抢占，资源请求者不能强行夺取其他线程的资源，只能等其主动释放
  • 请求和保持，某个线程在请求其他资源的同时保持对现有资源的占有
  • 循环等待，T1占有T2的资源，T2占有T3的资源，T3占有T1的资源

上述四个条件都成立时就会形成死锁，缺一不可

wait()方法和notify()方法

这两个方法来自顶级类 Object

wait() 方法让一个线程处于等待状态，如果在调用这个方法时没有传入等待的时间，那么这个线程会处于无限期的等待 WAITING ，如果传入了时间，那么在时间到了之后还没有被唤醒那么它会自己醒来，然后进入就绪状态，注意这个方法在释放资源的同时也会将持有的锁也释放掉，和 sleep() 方法的区别就在这

提一下 yield() 方法，这个方法是会释放 cpu 的执行权，但是会保留执行的资格

notify() 方法，用于唤醒某个处于等待的线程，如果存在多个会随机唤醒一个，还有一个相似的 notifyAll() 方法，则是唤醒所有处于等待状态的线程

注意，比如某个线程通过某个锁调用了 wait() 方法使其变成等待状态，那么在唤醒这个线程的时候，也必须是持有这把锁的线程才可以进行唤醒否则会抛出 IllegalMonitorStateException 异常

LockSupport、Lock

Lock

Lock，是 java.util.concurrent.locks 里的一个接口

ReentrantLock 是它的一个实现

LockSupport

LockSupport 基本围绕着 park 和 unpark，park 的作用是给线程上锁，也就是阻塞，unpark 方法是释放锁

ReadWriteLock，读写锁

读操作可以很多线程一起读，但写操作同一时间只能有一个线程进行，且其他线程不能进行读和写

线程池

Java 通过 import java.util.concurrent.Executors 这个类来创建线程池，默认提供了四种，都是静态方法，直接 Executors.newXXXThreadPool 即可，比如 Executors.newCachedThreadPool() 创建一个缓存线程池

  • newCachedThreadPool ，创建一个可缓存的线程池，当线程长度超过了处理需要，可回收空闲线程，否则新建线程
  • newFixedThreadPool，创建一个定长的线程池，可用于控制并发数，多出的线程排队等候
  • newScheduledThreadPool，创建一个定长的线程池，支持定时及周期性任务执行
  • newSingleThreadExecutor，创建一个单线程的线程池，至多存在一个线程，可保证任务的执行顺序

线程的构造器

线程同步工具类

CountDownLatch

CountDownLatch ，倒计时器，适用于需要等待几个线程都执行完毕之后，再继续往下执行的场景

使用方法

通过 new 创建一个实例，传入线程数，然后在每个线程中调用其 countDown() 方法，调用一次就减1，直到等于 0，最后调用 await() 方法唤醒主线程

最后的执行结果一定是 t1,t2,t3 都执行完了才会执行最后的 System.out.println("线程都执行完了") 这句

但是注意，CountDownLatch 是一次性的用完就没了

CyclicBarrier

CyclicBarrier ，循环栅栏，和 CountDownLatch 类似，将一组线程阻塞到一个同步点，直到之后一个线程到达这个点，那么就会打开栅栏放行

使用方法

使用和 CountDownLatch 类似，只是构建时除了传入一个线程数量，还可以传入了一个 Runnable 接口，由于是函数式接口，所有可以直接使用 Lambda 表达式来写，当线程数量达到这个设定的线程数之后，会执行第二参数传入的方法

最后的执行结果也是等线程都完成了，才会执行第二参数传入的方法，不过和 CountDownLatch 除了构造方法不同之外，CyclicBarrier 是可以重复使用的，即上面代码的最后一部分，调用其 reset() 方法进行重置

Semaphore

Semaphore ，信号量，可用于控制同时的访问个数

使用方法

通过 new 构造，在构造是传入一个数字，表示可同时进行的线程数量，可用于限流

在每个线程执行前需要调用 acquire() 方法获得令牌，再开始执行线程体，用完了在调用 release() 方法释放令牌

最后的执行结果和上面的两个不一样，第一次因为一个线程都没有，会同时进入 构建时传入的数字 个，然后每当一个线程执行结束，就立马会有一个新的线程进入，以此往后
